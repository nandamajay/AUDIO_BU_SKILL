# Phase-2B Known Gaps

## WP11 candidate: --generate does not write artifact bytes to disk

### Symptom

`python3 -m orchestrator.main --onboard <target> --generate` completes
successfully and populates the `## Generation` section of
onboarding_report.md with path_hints, but writes zero artifact bytes
under generated/<run_id>/. The path_hint column becomes a promise the
orchestrator does not keep.

### Expected contract

- **PHASE2B_SPECIFICATION.md §5.1** (`docs/PHASE2B_SPECIFICATION.md:611`):
  > "Writes generated bytes to **an untracked directory** `generated/<run_id>/`."

- **PHASE2B_SPECIFICATION.md §5.4** (`docs/PHASE2B_SPECIFICATION.md:636`):
  > "The engine writes to `generated/<run_id>/` only. `generated/` is `.gitignore`'d (WP0.5)."

- **PHASE2B_SPECIFICATION.md §WP10 acceptance criterion** (`docs/PHASE2B_SPECIFICATION.md:429`):
  > "**Gate-open end-to-end:** Running with `--generate --target nord` produces the artifacts under `generated/<run_id>/`, the Phase-2B section in the report, and touches nothing under `arch/`, `sound/`, or any tracked kernel path."

- **runner.py module docstring — path-guard enforcement** (`orchestrator/generation/runner.py:38–44`):
  > "Path-guard enforcement (§5.4):
  >
  > `write_artifact_bytes(artifact, base_dir)` rejects any `path_hint`
  > outside `PATH_GUARD_ROOT`. Rejection returns `None`; the caller
  > replaces the `GeneratedArtifact` with a `GeneratorSkipped(
  > reason="path_guard_violation")`. The do_onboard dispatch calls
  > `sys.exit(1)` if any artifact in results carries that reason."

### Actual behavior

- `_run_generation` (`orchestrator/generation/runner.py:109–178`) fan-out
  loop at `runner.py:147–172` calls each generator, evaluates the path guard
  via `is_path_within_guard` (`runner.py:161`), converts rejections to
  `GeneratorSkipped(path_guard_violation)`, verifies via
  `verify_generation_result`, and stores
  `gc["generation"] = {"artifacts": [...], "post_verification": ...}`.
  No bytes are written to disk anywhere in this path.

- `write_artifact_bytes` (`orchestrator/generation/runner.py:86–106`) is
  defined, correct, and covered by
  `tests/test_generation_runner.py:207–250` (test 5 — unit-level; covers
  rejection paths (a)(b) and happy path (c), but not the end-to-end
  `do_onboard` write path) — but has **zero call sites in any production
  path**.

- `do_onboard` (`orchestrator/main.py:514–523`) calls `project_facts` and
  `_run_generation` only; it does not call `write_artifact_bytes` and does
  not implement the `sys.exit(1)` guard for `path_guard_violation` described
  in `runner.py:38–44`.

### Impact

- Users running `--generate` see path_hints in the report but no on-disk
  artifacts. The report becomes a false attestation.
- Downstream approval / materialization workflow is undefined.
- Manual verification of generated `.dtsi`, codec stub, and machine driver
  artifacts is blocked — reviewers have nothing on disk to inspect.
- WP10 acceptance criterion at `docs/PHASE2B_SPECIFICATION.md:429` is not
  satisfied.
- `path_guard_violation` is silently non-fatal today (spec says
  `sys.exit(1)`), which weakens the trust boundary.

### Proposed scope for WP11

- Wire `write_artifact_bytes` into `do_onboard` after `_run_generation`
  succeeds, iterating `gc["generation"]["artifacts"]` and calling
  `write_artifact_bytes` for each `GeneratedArtifact`.
- Construct `generated_run_dir` from the current `run_id` and pass it to
  `write_artifact_bytes`. Path-guard every write under `PATH_GUARD_ROOT`
  (already enforced by `write_artifact_bytes`).
- Implement the `sys.exit(1)` guard on `path_guard_violation` as described
  in `runner.py:38–44` — writes fail closed, not silently.
- Add `--dry-run` flag to preserve current report-only behavior for review
  workflows where side-effect-free execution is required.
- Extend test coverage (net-new file `tests/test_wp11_disk_writes.py` or
  extend `tests/test_generation_runner.py`) with a **happy-path** test that
  asserts bytes-on-disk under `generated/<run_id>/` for at least the
  `audioreach_topology` generator. Current test 5 covers only the rejection
  path; shipping WP11 without a success-path assertion would repeat the
  "function exists but not verified end-to-end" pattern that produced this
  gap.
- Update `docs/PHASE2B_SPECIFICATION.md` §5.1 / §5.4 / spec:429 with any
  contract refinements that emerge during implementation (e.g., `--dry-run`
  semantics, `run_dir` layout).

### Design constraints (must hold in WP11)

- **Writes gate report rendering, not the reverse.** Artifact bytes must
  land on disk before the `## Generation` section is rendered into
  `onboarding_report.md`. Otherwise the `path_hint` column can reference
  paths that never materialized. The current `_run_generation` → report
  render ordering must be inverted at the write step, or writes must be
  interposed between them.
- **Fail-closed on `path_guard_violation`.** The `sys.exit(1)` guard is
  non-negotiable; a violation is a trust-boundary breach, not a recoverable
  skip.
- **Idempotency.** Re-running `--generate` on the same `run_id` must either
  short-circuit (bytes already on disk, hashes match) or overwrite
  deterministically. Undefined behavior is not acceptable.
- **`--dry-run` must be truly side-effect-free.** No touches under
  `generated/`, no partial writes, no lock files.

### Blast radius

- `orchestrator/main.py`: `do_onboard` dispatch (`main.py:514–523`)
- `orchestrator/generation/runner.py`: `write_artifact_bytes` wiring and
  `sys.exit(1)` guard
- `tests/`: new happy-path fixture asserting bytes-on-disk
- `docs/PHASE2B_SPECIFICATION.md`: contract refinements if any

### Dependencies / risks

- **T4a/T5 gate interactions:** codec, DT, and machine-driver generators
  gate-skip on missing T4a/T5 rows in the current Nord snapshot. WP11 must
  not regress this gating behavior — writes only happen for
  `GeneratedArtifact` results, never for `GeneratorSkipped`.
- **`PATH_GUARD_ROOT` concerns:** verify `PATH_GUARD_ROOT` (`"generated/"`)
  resolves to the correct absolute path under both the skill root and any
  downstream execution context (CI, developer laptop, Nord vs Eliza targets).
- **Nord vs Eliza differences:** Eliza T5 donor-rule work is still in flight
  (`docs/PHASE2A_KB_FOLLOWUPS.md`). WP11 happy-path test should pin to a
  target where at least one generator returns `GeneratedArtifact` reliably
  (Nord `audioreach_topology` is the current candidate).
