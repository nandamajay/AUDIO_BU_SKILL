## 0. Reading Guide

### Column definitions

| Column | Meaning |
|---|---|
| **Value** | The literal string, integer, or symbolic identifier emitted in the fixture |
| **Emitted at** | Fixture file + approximate location (label / line context) |
| **Authority source(s)** | Where the value ultimately came from (see §0.2) |
| **Phase of first discovery** | Which project phase first confirmed or produced this value |
| **Citation** | Canonical file+line anchor confirming the authority |
| **Consistency** | For dual-authority values: do the two sources agree? |

### Authority categories (§0.2)

| Code | Meaning |
|---|---|
| **IPCAT-DIRECT** | Live Phase-1 IPCAT call result (gpio_list_gpios_from_map, cores_list_core_instances, chipio_get_qups, chips_list_chips, swi_search_swi) |
| **IPCAT-DERIVED** | Derived from IPCAT catalog counts via WP-C cardinality cross-check |
| **SCHEMATIC** | Confirmed or attested by board schematic reference (LD20-94441 or equivalent) cited in patch or profile |
| **UPSTREAM-BINDING** | Linux upstream kernel source (binding doc, driver source, or header in linux-nord tree) |
| **NORD-PATCH** | linux-nord patch (0003-*.patch or 0004-*.patch) that Nord engineering authored |
| **KB-RULE** | Phase-2A knowledge-base rule in crossverify_config.py (t5.donor.firmware.sa8775p, etc.) |
| **FIXME-REVIEWER** | Not confirmed by any authority; emitted as an explicit placeholder with FIXME marker; reviewer must resolve |
| **INFERRED-ARCH** | Architectural reasoning from upstream docs / driver source; not a live IPCAT read |

### How to interpret FIXME rows

A value coded FIXME-REVIEWER is **not a hallucination in the traditional sense** — the generator explicitly marks it as a placeholder, does not assert it is correct, and emits a machine-parseable `FIXME(...)` comment. These rows appear in the hallucination audit (§8) but are not treated as undetected fabrications.

## 7. Hallucination Audit

A hallucination is defined as: **a value emitted without a traceable authority, asserted as correct (not marked FIXME), and not derivable from any cited source.**

### 7.1 Investigation

Every value in §§1–4 was traced to at least one of: a live IPCAT call result (Phase-1B/1C), a linux-nord patch, a pinctrl driver constant, or an upstream kernel header. The FIXME-REVIEWER values are explicitly not-asserted; they are not hallucinations.

**Special examination — `qcom,nord-iq10-sndcard` (decision A):**
This compatible string does not appear in any upstream source, patch, or IPCAT result. It was introduced as an INFERRED-ARCH design decision to avoid reusing the qcs9100 reference compatible (which would falsely imply Nord IQ-10 hardware compatibility). The 0004 patch uses `qcom,qcs9100-sndcard` as a first-pass; WP5 deliberately chose a Nord-specific string. The generator emits a partial-artifact NCC row (`sound_card.driver_match.nord_iq10`) acknowledging this has no upstream driver match. **Verdict: not a silent hallucination — the deviation from patch is documented and flagged via partial-artifact row.**

**Special examination — `0x30000000` reg base:**
This value is from the 0003 patch (NORD-PATCH authority). The conflict with Nord boot logs (`0x07000000`) is explicitly noted in both the patch and the generated fixture as a `FIXME(sa8797p-audio)`. The generator does not assert the value is correct; it surfaces the conflict. **Verdict: not a silent hallucination — conflict is explicitly disclosed.**

**Special examination — RPMHPD_LCX/LMX:**
These power domain references are from the 0003 patch (lemans port). The patch explicitly marks them as wrong for Nord (`qcom,nord-rpmhpd does NOT expose LCX/LMX`). The generator reproduces the FIXME comment verbatim. **Verdict: not a silent hallucination — error is inherited from patch and re-disclosed.**
